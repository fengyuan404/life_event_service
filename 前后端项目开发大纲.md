# 人生大事服务管理系统前后端开发计划

## Summary

- 开发一个课程设计级 Vue + Spring Boot 全栈系统，围绕现有 `life_event_service` 数据库完成核心业务闭环。
- 功能范围：简化登录、首页统计、家属管理、逝者管理、墓区/墓穴管理、租赁办理、缴费登记、祭扫预约、到期提醒、月度营收。
- 技术路线：后端 Spring Boot 3 + MyBatis-Plus + MySQL；前端 Vue 3 + Vite + Element Plus + Pinia + Axios。
- 开发目标：一次成型，优先保证能启动、能演示、能贴合数据库课设报告和 SQL 脚本。

## Key Changes / Architecture

- 项目结构：
  - `backend/`：Spring Boot 后端工程，端口 `8080`。
  - `frontend/`：Vue 前端工程，端口 `5173`。
  - 继续复用当前 SQL 文件中的数据库名 `life_event_service` 和 8 张核心表。
- 后端依赖：
  - Spring Boot Web、Validation、MyBatis-Plus、MySQL Connector/J、Lombok。
  - 不引入 Spring Security，避免过度复杂；用自定义拦截器实现简化登录校验。
- 前端依赖：
  - Vue 3、Vite、Vue Router、Pinia、Axios、Element Plus、ECharts。
  - PowerShell 下使用 `npm.cmd` 执行前端命令，避免 `npm.ps1` 执行策略问题。
- 登录方案：
  - `POST /api/auth/login` 使用 `staff.username` + `staff.password_hash` 校验。
  - 演示账号沿用 SQL 数据：`admin01/hash_admin01`、`recv01/hash_recv01`、`fin01/hash_fin01`。
  - 登录成功返回 UUID token、员工姓名、角色；前端保存到 Pinia + localStorage。
  - 后端通过 `X-Auth-Token` 请求头识别登录状态；重启后 token 失效，可重新登录。
- 通用接口格式：
  - 成功：`{ "code": 0, "message": "ok", "data": ... }`
  - 失败：`{ "code": 400, "message": "错误原因", "data": null }`
  - 分页：`{ "records": [], "total": 0, "page": 1, "size": 10 }`

## Backend Implementation

- 分层结构：
  - `controller`：REST API。
  - `service`：业务校验和流程编排。
  - `mapper`：MyBatis-Plus 数据访问。
  - `entity`：对应 8 张表。
  - `dto`：登录、分页查询、业务创建/更新请求。
  - `vo`：列表展示、统计卡片、详情页返回对象。
  - `config`：CORS、登录拦截器、统一异常处理。
- 核心业务规则：
  - 创建租赁时必须校验家属、逝者、墓穴、员工存在。
  - 创建租赁时墓穴必须为 `empty`，成功后更新为 `rented`。
  - 缴费必须关联已存在租赁订单，金额必须大于 0。
  - 祭扫预约人数必须大于 0，默认状态为 `booked`。
  - 删除采用业务安全策略：主要使用状态更新，避免破坏外键历史数据。
- 后端 API：
  - `POST /api/auth/login`：登录。
  - `POST /api/auth/logout`：退出。
  - `GET /api/dashboard/summary`：统计家属数、逝者数、空置墓穴数、有效租赁数、本月收入、待祭扫预约数。
  - `GET /api/dashboard/month-income`：月度营收折线/柱状图数据。
  - `GET /api/families`、`POST /api/families`、`PUT /api/families/{id}`、`DELETE /api/families/{id}`。
  - `GET /api/deceased`、`POST /api/deceased`、`PUT /api/deceased/{id}`。
  - `GET /api/grave-areas`、`POST /api/grave-areas`、`PUT /api/grave-areas/{id}`。
  - `GET /api/graves`、`POST /api/graves`、`PUT /api/graves/{id}`、`GET /api/graves/empty`。
  - `GET /api/rents`、`POST /api/rents`、`PUT /api/rents/{id}`、`GET /api/rents/expiring`。
  - `GET /api/payments`、`POST /api/payments`。
  - `GET /api/sacrifice-books`、`POST /api/sacrifice-books`、`PUT /api/sacrifice-books/{id}/checkin`、`PUT /api/sacrifice-books/{id}/cancel`。
  - `GET /api/staff`：员工下拉和列表查询。
- 权限控制：
  - `admin`：全部接口。
  - `receptionist`：家属、逝者、墓穴查询、租赁、祭扫预约。
  - `finance`：缴费、租赁查询、营收统计。
  - 权限不足返回 `403`。

## Frontend Implementation

- 页面结构：
  - `/login`：登录页。
  - `/`：后台布局，左侧菜单 + 顶部用户信息。
  - `/dashboard`：统计看板。
  - `/families`：家属管理。
  - `/deceased`：逝者档案。
  - `/graves`：墓区墓穴管理。
  - `/rents`：租赁办理与到期提醒。
  - `/payments`：缴费管理与营收查询。
  - `/sacrifice-books`：祭扫预约。
- 交互规范：
  - 所有列表页提供搜索、分页、新增、编辑、状态操作。
  - 表单统一使用 Element Plus Dialog + Form 校验。
  - 删除类操作使用二次确认；实际调用后端软删除或取消接口。
  - 租赁办理页提供下拉选择：家属、逝者、空置墓穴、经办员工。
  - 看板使用 ECharts 展示月度收入，卡片展示关键统计。
- 前端状态：
  - Pinia `authStore` 保存 token、staffName、role。
  - Axios 请求统一加 `X-Auth-Token`。
  - 路由守卫：未登录跳转 `/login`，角色不允许则隐藏菜单并阻止访问。

## Test Plan / Acceptance Criteria

- 启动验证：
  - 后端 `mvn spring-boot:run` 正常启动，访问 `http://localhost:8080/api/dashboard/summary` 未登录返回 `401`。
  - 前端 `npm.cmd install`、`npm.cmd run dev` 正常启动，访问 `http://localhost:5173` 打开登录页。
- 数据库验证：
  - 先执行现有 SQL 脚本创建 `life_event_service`。
  - 后端连接 MySQL 后，所有列表页能加载测试数据。
- 功能验收：
  - 使用 `admin01/hash_admin01` 登录后能访问全部菜单。
  - 能新增家属、逝者，查询空置墓穴，创建租赁后墓穴状态变为 `rented`。
  - 能为租赁订单登记缴费，并在月度营收图表中体现。
  - 能创建祭扫预约，执行核销和取消。
  - 财务账号只能看到缴费、租赁查询、营收统计相关页面。
- 质量要求：
  - 后端统一异常返回，不把 Java 堆栈暴露给前端。
  - 前端所有接口失败都有明确提示。
  - 表单必填项、金额、日期、人数等基础校验完整。
  - README 写清数据库初始化、后端启动、前端启动、演示账号。

## Assumptions

- MySQL 已安装并可用，数据库账号后续默认使用 `root`，密码写入 `backend/src/main/resources/application.yml`。
- 当前系统是课程设计演示项目，不做生产级密码加密、JWT 刷新、审计日志和复杂审批流。
- 后续实现时可以在当前目录新增 `backend/` 和 `frontend/`，不会覆盖已有报告和 SQL 文件。
- 前端 UI 以管理后台为主，追求清晰可演示，不做复杂视觉设计。
